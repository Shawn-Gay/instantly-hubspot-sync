TECHNICAL ARCHITECTURE & TROUBLESHOOTING
========================================

Boot Sequence
-------------
When the service starts (src/index.ts), it runs these steps in order:

  1. Load & validate configuration (config.ts reads env vars)
  2. Run database migrations (Drizzle ORM auto-migration)
  3. Reset stale jobs (any jobs stuck in "processing" go back to "pending")
  4. Ensure HubSpot custom properties exist (creates group + 11 properties)
  5. Auto-register Instantly webhooks (if AUTO_REGISTER_WEBHOOKS=true)
  6. Start sync engine (processes queue every SYNC_INTERVAL_MS)
  7. Start poller (polls Instantly every POLL_INTERVAL_MS)
  8. Start HTTP server on PORT

If any step before the server start fails, the process exits with
code 1. Webhook registration failure is non-fatal (logged as error
but boot continues).


Database Tables
---------------
All tables are defined in src/db/schema.ts and managed by Drizzle ORM.

1. processed_events -- Deduplication tracking
   Column        Type             Notes
   ------        ----             -----
   id            integer (PK)     Auto-generated identity
   event_id      text             Unique index for dedup lookups
   event_type    text             e.g. "email_sent", "email_opened"
   lead_email    text             Email address of the lead
   campaign_id   text             Instantly campaign UUID (nullable)
   created_at    timestamptz      Default now()

2. sync_jobs -- Persistent job queue
   Column        Type             Notes
   ------        ----             -----
   id            integer (PK)     Auto-generated identity
   lead_email    text             Email address of the lead
   event_type    text             Event that triggered this job
   payload       jsonb            Full event data for HubSpot sync
   status        enum             pending | processing | completed | failed | dead_letter
   attempts      integer          Number of processing attempts (default 0)
   next_retry_at timestamptz      When to retry a failed job (nullable)
   created_at    timestamptz      Default now()
   updated_at    timestamptz      Default now()
   Index: (status, next_retry_at) for efficient job dequeuing

3. lead_contact_map -- Email-to-HubSpot contact cache
   Column            Type           Notes
   ------            ----           -----
   id                integer (PK)   Auto-generated identity
   lead_email        text           Unique index
   hubspot_contact_id text          HubSpot contact record ID
   updated_at        timestamptz    Default now()

4. poll_state -- Poller cursor tracking
   Column        Type             Notes
   ------        ----             -----
   id            integer (PK)     Auto-generated identity
   poll_type     text             Unique identifier for the poll type
   cursor        text             Pagination cursor (nullable)
   last_poll_at  timestamptz      When the last poll completed (nullable)
   updated_at    timestamptz      Default now()

5. sync_errors -- Error log for failed jobs
   Column            Type           Notes
   ------            ----           -----
   id                integer (PK)   Auto-generated identity
   sync_job_id       integer (FK)   References sync_jobs.id
   error_message     text           Error description
   request_payload   jsonb          What was sent (nullable)
   response_payload  jsonb          What came back (nullable)
   created_at        timestamptz    Default now()


Queue System
------------
The sync_jobs table acts as a persistent job queue. Here's how jobs
flow through it:

  Status Flow:
    pending --> processing --> completed
                    |
                    v
                 failed --> (retry) --> pending ...
                    |
                    v (after 5 attempts)
               dead_letter

  Processing:
    - The sync engine runs every SYNC_INTERVAL_MS (default 10s)
    - Each cycle dequeues up to 10 jobs (pending, or failed with
      next_retry_at <= now)
    - Jobs are marked "processing" before work begins
    - On success, all jobs in the batch are marked "completed"
    - On failure, each job is individually marked "failed"

  Retry Logic (exponential backoff):
    Attempt 1 fails -> retry after 30 seconds
    Attempt 2 fails -> retry after 60 seconds
    Attempt 3 fails -> retry after 2 minutes
    Attempt 4 fails -> retry after 4 minutes
    Attempt 5 fails -> moved to dead_letter (no more retries)

    Formula: backoff = 30s * 2^(attempt - 1)

  Crash Recovery:
    On startup, any jobs stuck in "processing" (from a previous
    crash) are reset to "pending" so they get retried.


Rate Limiting
-------------
Token-bucket rate limiters prevent hitting API limits:

  HubSpot:    90 requests per 10 seconds (conservative)
  Instantly:  10 requests per 10 seconds (conservative)

The limiters block (sleep) rather than throw errors, so requests
are automatically throttled without losing data.


API Endpoints
-------------

POST /webhooks/instantly
  Receives webhook events from Instantly. Always returns 200 to
  prevent Instantly retry storms. Events are deduplicated and
  queued for processing.

  Request: Instantly webhook payload (JSON)
  Response:
    { "status": "ok", "processed": true, "duplicate": false }
    { "status": "ok", "processed": false, "duplicate": true }
    { "status": "ok", "processed": false, "error": "processing_failed" }

GET /health
  Returns service status and queue statistics.

  Response (healthy):
    {
      "status": "ok",
      "timestamp": "2025-01-01T00:00:00.000Z",
      "queue": {
        "pending": 3,
        "completed": 150,
        "failed": 1
      }
    }

  Response (database unreachable):
    HTTP 503
    { "status": "error", "message": "Database unreachable" }

GET /
  Redirects to /health.


Troubleshooting
---------------

Checking Queue Stats:
  curl https://your-app.up.railway.app/health

  Look at the "queue" object. Key things to watch:
    - "pending" growing = sync engine may be stuck or slow
    - "failed" count = jobs are failing but will be retried
    - "dead_letter" count = jobs that failed 5 times, need attention

Investigating Dead Letters:
  Connect to the database and query:

    SELECT sj.id, sj.lead_email, sj.event_type, sj.attempts,
           se.error_message, se.created_at
    FROM sync_jobs sj
    JOIN sync_errors se ON se.sync_job_id = sj.id
    WHERE sj.status = 'dead_letter'
    ORDER BY sj.created_at DESC;

  Common causes:
    - HubSpot access token expired or revoked
    - HubSpot API rate limit exceeded (shouldn't happen with limiter)
    - Malformed event payload from Instantly

Reprocessing Dead Letters:
  To retry dead-lettered jobs, reset them to pending:

    UPDATE sync_jobs
    SET status = 'pending', attempts = 0, next_retry_at = NULL
    WHERE status = 'dead_letter';

Common Errors:

  "Missing required environment variable: X"
    A required env var is not set. Check your .env file.

  "Instantly API error: 401"
    Your INSTANTLY_API_KEY is invalid or expired.

  "Instantly API error: 429"
    Rate limited by Instantly. The built-in limiter should prevent
    this. If it happens, consider increasing the poll interval.

  "Health check failed"
    The database is unreachable. Check DATABASE_URL and that
    PostgreSQL is running.

  "Fatal boot error"
    Something failed during startup. Check the logs for the
    specific step that failed (migration, property creation, etc.).

Logs:
  The service uses structured JSON logging. All log entries include
  a timestamp and context fields. In Railway, logs are viewable in
  the deployment dashboard.
